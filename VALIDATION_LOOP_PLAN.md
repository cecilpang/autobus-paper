# Neuro-Symbolic Validation Loop: Execution Plan

## 1. Overview
This document outlines the planned upgrade for the `autobus-prolog` skill. The upgrade transitions the skill from a linear script into a robust **Neuro-Symbolic Reasoning Loop**. This multi-agent "Chain of Thought" with "Consensus Validation" ensures the generated Prolog logic is syntactically valid, schema-compliant, and strictly adheres to the business requirements before execution.

## 2. Architecture & Roles (The "Lead-Validator" Pattern)

The process is orchestrated through a State-Based Architecture utilizing versioned run histories.

### The Personas
*   **Lead Agent (Gemini CLI Orchestrator):** Manages the filesystem state, triggers sub-agents, merges human feedback into task instructions, and handles the final human-in-the-loop approval.
*   **Generator Sub-Agent:** Specialized in generating Janus-SWI Prolog logic that maps natural language requirements to the SQLite database schema.
*   **Evaluator Sub-Agents (x3):** Peer reviewers running in parallel. They act as independent auditors. To ensure model diversity, these evaluators will utilize different LLM backends (or distinct heuristic expert personas if backend switching is constrained).
    *   **Evaluator A:** Focuses on Business Logic & Requirements Equivalence.
    *   **Evaluator B:** Focuses on Janus-SWI Syntax & Database Schema Mapping.
    *   **Evaluator C:** Focuses on Edge Cases & Security (SQL Injection/Safety).
*   **Reviewer Sub-Agent:** Synthesizes the three independent peer reviews. It identifies contradictions, aggregates findings, and prepares the final report.

### State Persistence Directory Structure
All intermediate artifacts are stored to enable traceability and backtracking:
```text
generated/runs/<task_id>/
├── v1/
│   ├── candidate.pl           # Generated by the Generator
│   ├── eval_A_report.md       # From Evaluator A
│   ├── eval_B_report.md       # From Evaluator B
│   ├── eval_C_report.md       # From Evaluator C
│   └── review_summary.md      # Aggregated by Reviewer
├── v2/                        # Created if human provides feedback
│   └── ... 
└── current/                   # Symbolic link or pointer to the active version
```

## 3. Workflow Steps

1.  **Context Loading:** 
    *   Lead Agent loads the target `Task_<ID>.md`, the database schema, and the Prolog template.
2.  **Generation:** 
    *   Lead Agent spawns the **Generator Sub-Agent**.
    *   Output: `generated/runs/<task_id>/v<N>/candidate.pl`
3.  **Parallel Evaluation:**
    *   Lead Agent spawns the three **Evaluator Sub-Agents** concurrently.
    *   Each evaluator independently analyzes `candidate.pl` against the task instructions and runs a syntactic compilation check (`swipl -c`).
    *   Outputs: `eval_A_report.md`, `eval_B_report.md`, `eval_C_report.md`
4.  **Consensus Review:**
    *   Lead Agent spawns the **Reviewer Sub-Agent**.
    *   The Reviewer reads the three evaluation reports, resolves or highlights "split decisions" (e.g., A says line 10 is an error, B says it is correct), and documents observations.
    *   Output: `review_summary.md`
5.  **Human-in-the-Loop Approval:**
    *   Lead Agent presents `review_summary.md` and `candidate.pl` to the human user for a final decision.
6.  **Resolution (Branching):**
    *   **If Approved:** Lead Agent promotes `candidate.pl` to `generated/<task_id>_logic.pl` and executes it via the bridge script.
    *   **If Feedback Provided:** Lead Agent merges the human feedback with the original instructions, increments the version directory (e.g., `v2/`), and restarts the loop at Step 2.

## 4. Supporting Tooling

To facilitate the syntactic checks by the evaluators, a new helper script will be created:
*   **File:** `autobus-prolog/scripts/validate_syntax.py`
*   **Purpose:** A lightweight wrapper around the `swipl -c <file.pl>` shell command. It provides a clean, machine-readable success/failure output for the sub-agents to consume during their evaluation phase.
